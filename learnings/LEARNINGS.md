## LRN-20260319-032

- **日期**：2026-03-19
- **优先级**：high
- **状态**：pending
- **类型**：最佳实践
- **范围**：架构设计 / 白纱
- **来源**：Gemini handoff — Minisforum PVE 蓝图（2026-03-19）
- **触发原话**：老板："嗯嗯，学到东西就完整的记录起来，以后就可以更好的帮我出主意啦"
- **教训（4条）**：

  **① 哨兵营 = 有执行权的守卫，不只是报警器**
  监控系统必须有"主动掐断"的能力（如掐断 API Key、触发硬止损），而不只是发 Telegram 报警等人来处理。黑天鹅来了人来不及处理，系统必须能自己动手。下次给老板出风控方案时，哨兵的职责 = 监控 + 执行，不只是监控。

  **② 基础设施隔离 > 代码熔断**
  PVE 虚拟化让 VM 4（进化树）爆内存，VM 1/VM 2（实盘）完全无感。这是 hypervisor 层的硬隔离，比代码层的 try/except / Circuit Breaker 可靠一个量级。出方案时要分清楚：代码熔断是"软保护"，基础设施隔离是"硬保护"，有条件要优先用硬保护。

  **③ ZMQ 架构天然横向扩展：单机→多 VM 只改一个 IP**
  单机 `tcp://127.0.0.1:5555` 到 PVE 多 VM `tcp://192.168.1.11:5555`，代码零改动，架构升级一个量级。这验证了 Feed Gateway 选 ZMQ 的决策是正确的。Nitro → Minisforum 迁移成本几乎为零。

  **④ 双盘策略 = 按用途分而非按容量分**
  系统盘选"稳定优先"（Samsung 990 Pro），数据盘选"TBW/IOPS优先"（WD SN850X）。不是为了备份，是因为热路径（高频写入日志/Tick）和冷路径（系统文件）对存储的诉求完全不同。设计存储方案时按用途隔离，不按容量叠加。

- **出现次数**：1
