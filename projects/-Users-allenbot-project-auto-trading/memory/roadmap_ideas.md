## 通讯部 — SQLite 全纪录方案（2026-03-03 无极+太极+白纱三方讨论）

### 讨论结果

**核心结论：SQLite 是"完美上下文"的基础设施**

无极原话："能不能做成种子？SQLite"、"那这个不就是等同完美上下文了？每一句话都钉下去，什么都可以提取，而且是3方会话全纪录，就不需要什么handoff checkpoint啦"

太极补充："白纱那个'单一权威来源'，SQLite就是答案"、"决策库最值得做，技术上最简单"

**白纱确认的逻辑**：
- 每句对话存进SQLite → 需要回顾时直接查 → 不受上下文窗口限制
- 窗口变成"工作记忆"，SQLite变成"长期记忆"，按需提取
- 不是"不需要CHECKPOINT"，而是CHECKPOINT变成自动生成的视图（SQL查出来），不再手工维护
- handoff同理：自动生成，不手写

**消耗对比（无极问）**：
- 现在翻文档：预先加载4个文件，很多内容用不到，消耗大
- SQLite按需查询：只返回查询结果，消耗更小，毫秒级
- 无极的直觉是对的：跟上下文窗口无关，直接查"最近N分钟的对话"精准提取

### 延伸问题（待讨论）
1. **搜索维度**：按时间（"最近5分钟"）够用吗？还需要按关键词/项目/人物筛选？
2. **表结构**：decisions / tasks / sessions 三张表够用吗？还需要什么？
3. **自动写入**：对话怎么自动存进SQLite？需要一个hook或者脚本？
4. **读取协议**：恢复协议改成"只查SQLite"，查什么字段、返回多少条？

### 建议表结构（初版）
| 表 | 存什么 |
|---|---|
| `decisions` | 无极所有拍板：时间/原话/决策内容/关联项目 |
| `tasks` | 主线支线状态：黑丝白纱各自在做什么/完成没 |
| `sessions` | 每次会话摘要：谁聊/聊了什么/关键结论 |
| `messages` | 每句对话完整原文：时间/说话者/内容 |

SQLite + 自动快照（每次更新后导出 dashboard_snapshot.md）= 最优组合。

## Intrabar Simulation（1m精度 + 1h策略）

**核心**：1h策略信号不变，止损/TTP追踪改用1m级别逐根K线判断，而不是1h K线的4个点。

**无极原话**："我先说说为什么会想到把1m放进来滚动，因为要模拟实盘。就是比如1h成绩好，但是自由开关高低4个数据，如果加了进来就能知道TTP是不是真的可以移动，这是第一点，用来验证TTP的。"

### 第一点：验证TTP不是回测幻觉
- 现状：1h回测里，TTP用4个价格点（OHLC）近似判断，很粗糙
- 加入1m后：1h内60根1m K线，TTP的激活/移动/触发都有精确的时间戳
- 验证：1h回测里说TTP触发了，1m精度下是真的触发了吗？还是那1小时内价格根本没到那个位置？
- **技术可行**：backtest.py外层1h循环，内层加60根1m循环，策略接口不变，只有止损/TTP切换到1m精度

### 第二点：宏观指标定义日线级别regime
- 12个宏观指标（MVRV/NUPL/Fear&Greed等）是日线数据，可以向下渗透给所有子周期打大方向标签
- 不替代ADX regime，而是两层配合：宏观定大方向（几个月的周期），ADX定局部时机（几天的波段）
- 注意：宏观指标历史数据长度需要核实（Fear&Greed从2018年开始，基本能对齐K线数据）

### 第三点：宏观指标照妖镜（用已知事实验证投票准确率）
- 用种子v3 PnL反推"已知事实"（哪段时间做多策略净赚→事实牛市，做空净赚→事实熊市）
- 把12个宏观指标历史读数和已知事实对比，算各指标的准确率
- **目的**：找出哪2~3个指标最靠谱、最优阈值是多少 → 把主观设权重的MarketScore改成数据驱动
- **注意**：已知事实只能回看，不能实盘用；宏观指标是实盘时唯一能实时读取的工具

## 用已知事实替代ADX定义专考分组（新核心想法）

**无极原话**："既然知道了已知事实，为什么现在不把专考班级直接对应的贴上已知事实，然后进照妖镜呢？"

### 现状（人话）
专考的"牛程/熊程/盘整程"分组，是用ADX打的标签：
- 种子v3生成时，每笔交易开仓时查ADX值 → 打上 bull/bear/ranging 标签（存在 `regime_at_open` 字段）
- 专考排名时，把每个策略的交易按这个标签分堆，分别算PnL
- 问题：ADX打的标签可能是错的——它只是一个技术指标推断，不是"事实"

### 无极的想法
- 既然种子v3里知道了每段时间哪类策略真的赚钱，为什么不用这个来定义牛熊？
- 把"已知事实regime"贴到专考上，让排名基于真实的市场状态分类，而不是ADX推断

### 技术实现思路（待讨论）
- 从种子v3所有策略，对每个时间段算：做多策略平均PnL vs 做空策略平均PnL vs 震荡策略平均PnL
- 哪类净赚最多，那段时间就是对应的regime（注意：用全体策略的平均，避免循环依赖）
- 或者更简单：用BTC价格涨跌幅定义（30天涨>20% = 牛，跌>20% = 熊，其余 = 盘整）——对所有策略独立
- 用这个新标签替代 `regime_at_open`，重新跑专考排名 → 进照妖镜

### 价值
- 现在的专考"牛程表现"建立在一个可能不准确的ADX分类上 → 结论可信度打折扣
- 如果用真实交易结果反推的regime重新分类，专考才是真正意义上的"在对的市场环境里找对的策略"

### 待解决问题
- 循环依赖风险：用策略PnL定义市场状态，再用市场状态评价策略 → 需要用全体策略平均或独立的价格走势来定义，不能用被评价的策略自己的PnL
- 时间窗口：每天/每周/每月定义一次regime？粒度影响结果
- 与ADX regime的对比：两套标签跑出来的专考排名差距有多大？

## 数据增强想法（2026-03-03 无极+黑丝讨论，记录备用）

> 无极明确指示："这些聊过的东西要记住呀，不然我会忘记的。。。那就先记起来，完整记录。然后就先不管了，目前按照我们自己的路线跑"

### 无极回测哲学（原话保留）
"回测只管根据历史数据测，不管实盘如何，我只要他能准确识别出周期就行了。再怎么说，历史数据也是真实的。这不是我们要担心的事。我不是未来人，不预测未来，只管做好本分去见证它。"

### 想法1: 5组滑窗 Regime WF（讨论后暂缓）
- 牛+初牛(9段) → 初牛+盘整(5段) → 初牛+盘整+初熊(11段) → 盘整+初熊(8段) → 初熊+熊(10段)
- 像滑动窗口，从牛端滑到熊端，过渡带被多角度检验
- 分析结论：牛单跑98.7%通过率，加了没用；5组比单regime更严但也更合理
- **暂缓原因**：最终采用更简单的"盘整ranking-only"方案，不需要复杂滑窗

### 想法2: 价格反转 / 涨变跌（记录，中长期）
- 无极原话："我记得看盘工具有一个功能是把数据的涨跌倒过来看，不过不是时间，只是涨变跌 跌变涨"
- 概念：K线数据的涨跌互换（价格反转），不改变时间顺序
- **可行性**：概念上合理，等于造一份"平行宇宙"验证数据
- **问题**：策略依赖绝对价格（MA20的值变了，信号时间点不同），必须拿反转后的K线重跑整个回测引擎
- **工程量**：84策略×13币×10周期=10,920组回测，约19GB新种子，与当初种子v3等量
- **结论**：想法有价值，但投入产出比暂时不划算。等主管线稳定后可考虑

### 想法3: ETH独立时间线（记录，中期）
- 无极原话："不如这样。我们正加一个币种，ETH，一样按照BTC的做法，做一套出来"
- 概念：为ETH做独立的regime时间线（类似BTC的21段），解决盘整数据不够的问题
- **可行性**：ETH与BTC高度同步但有差异点（2020 DeFi夏天/2022 Merge/ETH-BTC汇率独走）
- **价值**：可能多出1-2段差异化盘整/初牛，不止解决盘整问题，整体精度都提升
- **结论**：方向对，放后期做。等Gate管线稳定后可以给每个币做自己的regime时间线

### ❌ 已否决想法
- **对半切**：无极正确指出"一整段跑跑和一半一半的跑，会有差别？"——切成两半跑跟整段跑结论基本一样，不增加独立信息
- **时间倒带**：策略指标是往回看的（MA20看过去20根），倒带后指标看到的"过去"实际是真实时间的"未来"，等于偷看未来数据，结果不可信
