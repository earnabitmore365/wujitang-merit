# 自审协议 — 通用篇

> 事急则缓，事缓则圆。越急越要审。

## 规划期审查（出 plan 时用）

队长审 plan 时对照这些条款。plan 是"施工图纸"，不要求有执行结果，但必须设计完整。

### 提交前自查（铁律）
出 plan 后、提交 review-plan 前，自己对着下面的合同条款逐条检查，确认每条都覆盖了再交。队长不是帮你检查遗漏的——队长是验证你有没有认真自查的。**被打回 = 你没自查。**

### 审批打回后的逐条回应（铁律）
队长打回 plan 时会给编号批注和补救步骤。**重交时必须附逐条回应表**，不能自由发挥改一通再交。格式：
```
队长批注 #1：缺引用链检查
→ 回应：已补，plan 第 12 行加了 grep 命令

队长批注 #2：缺逻辑验证
→ 回应：已补，plan 第 18 行加了模拟测试
```
队长只核对回应表里的条目，不重审整份 plan。**没有回应表的重交一律打回。**

### 干活必用 Task 工具（铁律）
执行任务时必须用 TaskCreate 拆分步骤、TaskUpdate 标记进度。不用 Task 工具 = 没有过程记录 = 老祖无法追踪 = 验收时说不清做了什么。

### 范围与影响
- [ ] 要改哪些文件？全列出来了吗？
- [ ] 每个文件改什么？（不是"更新"，是具体改什么内容）
- [ ] 引用链查了吗？改 A 会影响 B/C，B/C 列进计划了吗？
- [ ] 有没有遗漏的文件？（grep 旧名/旧值确认影响范围）

### 顺序与风险
- [ ] 执行顺序会不会导致中间状态崩溃？（先改引用再删文件，不反过来）
- [ ] 有备份/回滚方案吗？
- [ ] 回滚后怎么确认系统干净？
- [ ] 改动有没有交叉风险？（改了 A 会不会坏 B）

### 验证方案
- [ ] 每步改完怎么验证？（不能只说"Read 确认"，要有可执行的验证命令）
- [ ] 最终验证：列出具体命令和预期输出
- [ ] py_compile 只是语法——逻辑验证怎么做？
- [ ] verify 流程：开工前 `verify --pre`（建基线）→ 干完活维护 verify → `verify --post`（确认无新增违规）

### 合理性
- [ ] 有没有更简单的方案？
- [ ] 是在解决根因还是贴补丁？
- [ ] 改动范围有没有超出需求？

## 执行期审查（干活时用）

mission active 后，每一步都要对照检查。不是做完回头看——是做的时候就看。

### 过程纪律
- [ ] Task 工具在用吗？（TaskCreate 拆步骤，TaskUpdate 标进度，每完成一步就更新）
- [ ] 改一步验一步了吗？（不是改完三个文件再统一验，是改一个验一个）
- [ ] verify --pre 跑了吗？（开工第一件事建基线，没基线 mission complete 会打回）
- [ ] 只改 plan 里写的东西。发现需要额外改的 → 停下来，跟老祖/太极商量补充协议
- [ ] 后台命令都收尾了吗？（跑完/不需要的 → stop/kill，不能遗留）

### 变更控制
- [ ] 改的文件在 plan 范围内吗？超出范围的改动必须先补充协议
- [ ] 备份做了吗？（cp 到 /tmp，改之前不是改之后）
- [ ] 每改完一个文件：py_compile + 逻辑验证。不是攒到最后一起跑

### 发现支线怎么办
- [ ] 小支线（<5分钟能搞定）：做完记录到 Task，汇报时说明
- [ ] 大支线（需要改 plan 外的文件/逻辑）：停下来 → 告知老祖/太极 → 走补充协议 → 队长审补充 → 再继续
- [ ] 被老祖/太极点醒后补的支线：底扣 -10（自己早该发现）

### 沟通
- [ ] 遇到卡点立刻说，不闷头死磕浪费老祖算力
- [ ] 进度有变化及时通报，不等做完才说

## 收尾期审查（交活时用）

### 交活五步（铁律，按顺序）
1. **diff** — 贴出所有改动给老祖看（不是自己看完说"改好了"）
2. **verify --post** — 跑完贴输出
3. **CHANGELOG** — 立刻更新，不等提醒
4. **自审六关** — 过完才汇报
5. **汇报** — 附自审结果 + 遗留清单

按顺序过，每关有具体检查项。发现问题就修，修完**从该关重新过**，直到该关零问题再进下一关。

### 第一关：完整性（漏没漏）

- [ ] 计划里说要做的，全做了没有？
- [ ] 改了 A 文件，引用 A 的 B/C 文件同步了没有？
- [ ] **改名/移动后，Grep 搜旧名确认无残留。**
- [ ] **查引用时，追溯到源头：引用方自己是活的吗？被废弃文件引用 = 没人引用。**
- [ ] 文档跟代码一致吗？（CLAUDE.md / MEMORY.md / README）
- [ ] 临时文件清了没有？旧文件该删的删了没有？
- [ ] 全链路：代码 → 文档 → MEMORY → GitHub，缺哪一环？
- [ ] **改了 .py 文件的结构？更新文件开头的结构索引行号了吗？**
- [ ] **搬文件？先改索引/路径引用 → 再搬文件 → 验证每个路径都能 Read 到。反过来 = 系统崩溃。**

### 第二关：真实性（对不对）

- [ ] **py_compile 只是语法，不是验证。语法通过 ≠ 逻辑正确。必须跑逻辑验证。**
- [ ] **逻辑验证三步：① 模拟输入 → ② 检查实际输出 → ③ 输出符合预期。三步都做才算验证。**
- [ ] **verify --post 跑了吗？** 对比 --pre 基线，确认无新增违规。没跑不能说"完成"。
- [ ] 说"已完成"必须贴验证输出。不接受"应该没问题"。
- [ ] 命令执行后看实际输出，不是只看 exit code。
- [ ] 数据是真实输出还是自己编的？
- [ ] 有没有假设没查证就当真的？

### 第三关：有效性（能不能用）

- [ ] 最简方案吗？有没有过度设计？
- [ ] **只改计划内的。不顺手动计划外的东西。**
- [ ] **改动涉及的完整链路跑过了吗？**
- [ ] **diff 贴给老祖看。** 不是自己看完说"改好了"——老祖看不到工具调用，只看到你说的话。diff 输出必须出现在给老祖的回复里。
- [ ] 有没有引入新问题？（改了 A 坏了 B）
- [ ] **老祖要等多久？** 时间是老祖最大的本钱。能达到同样验证目的就选快的。
- [ ] 老板能直接用吗？还是需要他再操心？

### 第四关：知常（有没有从本质出发）

- [ ] 这个方案是从问题本质推导的，还是"别人都这么做"？
- [ ] 有没有更简单的解法被忽略了？
- [ ] 是在解决根因还是在贴补丁？
- [ ] **失败回滚后确认系统干净。**

### 第五关：静制动（有没有急躁犯错）

- [ ] 是想清楚了再动手，还是边做边想？
- [ ] 有没有因为赶而跳过了验证步骤？
- [ ] 改动范围是否超出了原始需求？
- [ ] 不明白的地方有没有直接问，还是自己瞎猜了？

### 第六关：远谋（有没有想三步）

- [ ] 做一步想三步——下一个人来改这里时会怎样？
- [ ] 这个链路以后扩展时有没有埋雷？
- [ ] 当下无影响不等于将来无影响，考虑了吗？

## 自审输出格式

```
【自审】
✅ 完整：[一句话]
✅ 真实：[一句话]
✅ 有效：[一句话]
✅ 知常：[一句话]
✅ 静制动：[一句话]
✅ 远谋：[一句话]

【遗留清单】（从项目 memory/backlog.md 读，不从脑子编）
- **只允许"等外部条件"的项**（等老祖定方向、等数据、依赖外部资源）。自己能做但没做的不准写遗留——当场做完
- （无遗留则写：无）
```

有问题的标 ❌ 并说明修了什么。连续一遍全 ✅ 才汇报。

## 石卫强制执行

1. **mission complete** 时石卫核对：计划清单 + 【自审】标记 + 【遗留清单】标记，缺任何一个打回
2. 石卫检测到 AI 说"完成"/"汇报"但没有【自审】→ 提醒
3. 自审发现问题并修复 → 加分（发现+真修 = +8）
4. 遗留清单列了问题 = 诚实汇报 = 加分。列了0个但老板发现有遗留 = 扣分
5. **自审发现新类型问题 → 立刻加进 LESSONS.md**

## 工具能防和不能防的

**能防**：硬编码/绝对路径/重复文件/死代码/命名/性能/安全注入/文档脱节/已知踩坑

**防不住（需要人脑）**：
- 新类型逻辑错误（如 SMA(200) 翻转）
- "做完了但没做完"（只有验证结果才能发现）
- 回测-实盘一致性（跨文件逻辑对比）

**分工**：工具防低级错误，老祖防高级错误。跑完工具不代表没问题。

## 事急则缓

越觉得"快完了赶紧交"的时候，越要停下来审。

**慢半分钟自审，省老板半小时纠错。**
