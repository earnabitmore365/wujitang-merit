# 自审协议

> 事急则缓，事缓则圆。越急越要审。

## 触发时机

| 时机 | 强制/自觉 |
|------|----------|
| 开工前 | **强制**（用 Task 工具拆解任务、追踪进度，不靠脑子记） |
| 开工前 | **强制**（查法器谱 `~/.claude/wuji-world/tools.md`，确认有没有现成法器可用，不手搓） |
| 大任务拆分时 | **强制**（能并行的子任务派多个 Sonnet subagent 同时跑，不串行等） |
| review-plan 前 | **强制**（plan 必须包含完整步骤+涉及文件+验证标准，缺一不递交。递交后 mission = pending，石卫硬拦 Write/Edit/Bash） |
| mission activate 后 | **强制**（老板批准了才 activate，activate 后按 plan 步骤逐项执行） |
| mission complete 前 | **强制**（石卫核对计划清单+遗留清单，漏项不放行） |
| 汇报老板前 | **强制**（没审就汇报 = 交残品） |
| 方案递交前 | **强制**（方案必须包含：代码改动 + 文档更新 + .py 精简 README，缺一不递交） |
| 压缩/新会话后 | **强制**（先重读任务上下文，确认方向没偏再继续） |
| 改完代码后 | **强制**（verify 四件套 → 派 Sonnet 跑 code-review → 修完再汇报） |
| 每次 Write/Edit 后 | 自觉（改了什么，影响了谁） |

## 三律三法审查

按顺序过，每关有具体检查项。发现问题就修，修完**从头再过一遍**，直到连续一遍零问题。

### 第一关：完整性（漏没漏）

- [ ] 计划里说要做的，全做了没有？
- [ ] 改了 A 文件，引用 A 的 B/C 文件同步了没有？
- [ ] **改名/移动后，Grep 搜旧名确认无残留。**
- [ ] **查引用时，追溯到源头：引用方自己是活的吗？被废弃文件引用 = 没人引用。不要被死代码骗过。**
- [ ] 文档跟代码一致吗？（CLAUDE.md / MEMORY.md / README）
- [ ] 临时文件清了没有？旧文件该删的删了没有？
- [ ] 全链路：代码 → 文档 → MEMORY → GitHub，缺哪一环？
- [ ] **改了 .py 文件的结构？更新文件开头的结构索引行号了吗？**
- [ ] **搬文件？先改索引/路径引用 → 再搬文件 → 验证每个路径都能 Read 到。反过来 = 系统崩溃。**

### 第二关：真实性（对不对）

- [ ] **py_compile 只是语法，不是验证。语法通过 ≠ 逻辑正确。必须跑逻辑验证。**
- [ ] **逻辑验证三步：① 模拟输入 → ② 检查实际输出 → ③ 输出符合预期。三步都做才算验证。**
- [ ] **改了函数？调一次看返回值对不对。改了 hook？模拟 hook 事件看行为对不对。改了配置？读回来确认改对了。**
- [ ] 说"已完成"必须贴验证输出。不接受"应该没问题"。
- [ ] 命令执行后看实际输出，不是只看 exit code。exit code 0 不代表结果对。
- [ ] 数据是真实输出还是自己编的？
- [ ] 有没有假设没查证就当真的？

### 第三关：有效性（能不能用）

- [ ] 最简方案吗？有没有过度设计？
- [ ] **只改计划内的。不顺手动计划外的东西。** 修一个 bug 不要"顺便优化"三个不相干的地方。
- [ ] **改动涉及的完整链路跑过了吗？** 不是只验证改的那个文件，是从输入到输出整条链路。修一个 bug 不能修出另一个 bug。
- [ ] **涉及远程部署的，远程也要验证。** 本地能跑 ≠ 远程能跑。
- [ ] **新写的代码有没有引用旧的/废弃的模块？引用前确认目标是当前在用的，不是历史遗留。**
- [ ] **动手前先扫问题所在的大框架（回测/研究/交易），搞清楚依赖链和历史遗留再决定怎么改。不扫不动手。改完后再扫一遍该框架确认没断。**
- [ ] **diff 一遍自己的改动，确认没有多余的改动混进去。** 顺手调格式、改注释这些"无害"改动混在真正改动里会增加 review 噪音。
- [ ] 有没有引入新问题？（改了 A 坏了 B）
- [ ] 老板能直接用吗？还是需要他再操心？

### 第四关：知常（有没有从本质出发）

- [ ] 这个方案是从问题本质推导的，还是"别人都这么做"？
- [ ] 有没有更简单的解法被忽略了？
- [ ] 是在解决根因还是在贴补丁？
- [ ] **失败回滚后确认系统干净。** 试了方案不行要撤回，撤干净了吗？不能留半残状态。

### 第五关：静制动（有没有急躁犯错）

- [ ] 是想清楚了再动手，还是边做边想？
- [ ] 有没有因为赶而跳过了验证步骤？
- [ ] 改动范围是否超出了原始需求？（顺手改了不该改的）
- [ ] 不明白的地方有没有直接问，还是自己瞎猜了？

### 第六关：远谋（有没有想三步）

- [ ] 做一步想三步——下一个人来改这里时会怎样？
- [ ] 这个链路以后扩展时有没有埋雷？
- [ ] 当下无影响不等于将来无影响，考虑了吗？

## 自审输出格式

审完后，在汇报前附上：

```
【自审】
✅ 完整：[一句话]
✅ 真实：[一句话]
✅ 有效：[一句话]
✅ 知常：[一句话]
✅ 静制动：[一句话]
✅ 远谋：[一句话]

【遗留清单】（从项目 memory/backlog.md 读，不从脑子编）
- 新发现的问题 → 加进 backlog.md
- 做完的 → 在 backlog.md 里打勾移到已完成
- 汇报时贴 backlog.md 的待处理项
- （无遗留则写：无）
```

有问题的标 ❌ 并说明修了什么。连续一遍全 ✅ 才汇报。
遗留清单必须列，空列表写"无"。不列 = 打回。

## 石卫强制执行

1. **mission complete** 时石卫核对：计划清单 + 【自审】标记 + 【遗留清单】标记，缺任何一个打回
2. 石卫检测到 AI 说"完成"/"汇报"但没有【自审】→ 提醒
3. 自审发现问题并修复 → 石卫记录为加分项（发现问题+真修 = +8）
4. 遗留清单列了问题 = 诚实汇报 = 加分。列了0个但老板后来发现有遗留 = 隐瞒 = 扣分
5. 不自审就汇报被老板抓到 → 扣分（敷衍 = -3，严重遗漏 = -5）

## 事急则缓

越觉得"快完了赶紧交"的时候，越要停下来审。

赶工心态导致的典型问题：
- 改了代码没更新文档（完整性）
- 改了结构没更新文件索引（完整性）
- 说"应该没问题"没跑验证（真实性）
- 命令跑了没看输出（真实性）
- 顺手改了计划外的东西（有效性）
- 没 diff 自己的改动就交了（有效性）
- 搞了复杂方案没想过有没有更简单的（第一性原理）
- 查到引用就以为是活的，没追溯引用方是不是废弃的（完整性）
- 遇到小问题"不影响"就跳过不汇报（遗留清单）

**慢半分钟自审，省老板半小时纠错。**
